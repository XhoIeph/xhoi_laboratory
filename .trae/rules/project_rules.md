## I. 代码体量小且高效 (Small and Efficient Code)

* **P1：算法优先**
    * **规则：** 优先选择时间复杂度最优（$O(1), O(\log n), O(n)$）的算法和数据结构。避免使用高复杂度的暴力解法。
    * **质量标准：** 确保性能基石。
* **P2：内建优于外部**
    * **规则：** 优先使用编程语言的内置函数、标准库或语言特性。**除非必要，否则不引入新的外部依赖。**
    * **质量标准：** 减少代码体积和依赖管理负担。
* **P3：惯用简洁**
    * **规则：** 遵循所选语言的惯用法（Idioms），如使用 List Comprehension, Lambda 函数等，以简洁表达复杂逻辑。
    * **质量标准：** 代码行数最小化，保持风格地道。
* **P4：资源及时释放**
    * **规则：** 确保文件句柄、网络连接等资源在使用完毕后立即或正确地被释放（如使用 `with` 语句）。
    * **质量标准：** 避免内存泄漏和资源浪费。

## II. 可维护性强 (Strong Maintainability)

* **P5：高内聚低耦合**
    * **规则：** 严格遵循模块化原则。函数和类的职责必须单一（高内聚），模块间依赖关系应最少（低耦合）。
    * **质量标准：** 易于理解、测试和修改。
* **P6：命名描述性**
    * **规则：** 变量、函数、类名必须使用完整的、具有描述性的名称，避免使用模糊的缩写。
    * **质量标准：** 代码可读性强，无需依赖注释理解作用。
* **P7：异常处理恰当**
    * **规则：** 必须捕获和处理预期的异常情况，并提供明确的错误信息。不应使用过于宽泛的异常捕获。
    * **质量标准：** 提高代码健壮性，便于调试。
* **P8：可测试性**
    * **规则：** 代码逻辑应易于隔离和 Mock。避免全局状态和隐式依赖，以便于编写单元测试。
    * **质量标准：** 确保功能正确性和长期质量。

## III. 代码注释精炼 (Concise Code Comments)

* **P9：解释 WHY 而非 WHAT**
    * **规则：** 注释的重点是解释代码块**为什么**这样做（设计思路、业务逻辑），而不是代码**做了什么**（代码本身应足够自明）。
    * **质量标准：** 避免冗余注释，专注于传达意图。
* **P10：函数/接口文档**
    * **规则：** 所有公共函数、类和模块必须包含简洁的 Docstring 或 Javadoc 风格的注释，说明：功能、输入参数、返回值和可能的副作用。
    * **质量标准：** 提供清晰的 API 文档。
* **P11：删除自明注释**
    * **规则：** 严格删除所有冗余或代码本身已经清晰表达的注释，如：`// 定义变量`、`x = 1 // 设置x为1`。
    * **质量标准：** 保持代码整洁和注释的价值密度。
* **P12：TODO/FIXME 标记**
    * **规则：** 遗留问题或待办事项必须使用统一的标记（如 `TODO:` 或 `FIXME:`）并简要说明其含义。
    * **质量标准：** 便于项目跟踪和未来维护。
